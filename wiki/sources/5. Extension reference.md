# Extension Reference

This source provides comprehensive documentation for the Gemini CLI extension system. It covers the complete `gemini extensions` command group for managing extensions from the terminal, the `gemini-extension.json` manifest format with full schema details, extension settings and configuration management, custom commands via TOML files, Policy Engine contributions, themes, and various extension components including hooks, agents, and skills.

## Key Topics Covered

The document details how to install extensions from GitHub or local paths using `gemini extensions install`, with options for specific git refs, auto-updates, and pre-release versions. It explains the uninstall, disable, enable, and update commands, as well as development workflows using `gemini extensions link` for symlink-based testing. Extension scoping is documented, showing how extensions can be disabled per user or workspace scope, with workspace settings taking precedence over extension settings on merge.

The manifest schema (`gemini-extension.json`) is fully documented with all fields: `name`, `version`, `description`, `mcpServers`, `contextFileName`, `excludeTools`, `migratedTo`, and `plan.directory`. The settings system allows extensions to request user configuration like API keys, with sensitive values stored in the system keychain. Custom commands use a TOML-based format with directory structure determining command names (e.g., `commands/gcs/sync.toml` becomes `/gcs:sync`). Policy Engine contributions are documented with a tiered system where extension policies run in tier 2, below user/admin but above defaults.

## Variables and Patterns

The source documents variable substitution patterns for extension portability: `${extensionPath}` for the extension directory, `${workspacePath}` for the current workspace, and `${/}` for the platform-specific path separator. Context file loading is covered, with fallback to `GEMINI.md` when `contextFileName` is not specified. Conflict resolution rules state that extension commands have lowest precedence and are prefixed with extension names using dot separators (e.g., `/gcp.deploy`) when conflicts occur.
