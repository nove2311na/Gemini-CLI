# File Management with Gemini CLI

This tutorial from geminicli.com provides a comprehensive guide to exploring, analyzing, and modifying codebases using Gemini CLI. It covers the complete workflow from providing context through file inclusion, discovering files via search tools, making modifications, reviewing changes, and verifying results.

## Core Workflow

The document outlines a five-stage workflow for file management:

1. **Read** — Provide context via `@file` syntax, multiple file references, or directories
2. **Explore** — Use natural language to find unknown files using glob and list_directory tools
3. **Modify** — Direct targeted edits with the replace tool or create new files with write_file
4. **Review** — Unified diff confirmation before applying any changes
5. **Verify** — Run tests to confirm correctness of modifications

## File Context Injection

The `@` symbol forces Gemini CLI to read specific files and inject their content into the conversation context. This enables targeted analysis of known files. Users can chain multiple `@` references for multi-file context, or include entire directories for broad refactoring tasks. The document notes that directory inclusions consume more tokens, making this an important efficiency consideration.

## Exploration Tools

When users do not know exact file paths, Gemini CLI can search the project structure using glob or list_directory tools. This enables navigation of unfamiliar codebases through natural language queries like "Find the file that defines the UserProfile component." The agent returns specific paths that can then be used with `@` for subsequent operations.

## Modification Tools

The replace tool proposes targeted code changes rather than simple text replacement, enabling complex refactoring. The write_file tool creates new files from scratch, supporting complete folder structures. Both tools respect the safety-first approach of presenting changes for review before application.

## Safety Mechanism

Before any file modification, Gemini CLI presents a unified diff showing removed lines (red, prefixed with `-`) and added lines (green, prefixed with `+`). Users confirm with `y` or cancel with `n` to refine their prompt. This diff review interface serves as the primary safety mechanism for code changes.

## Privacy Control

Gemini CLI respects `.gitignore` by default, excluding node_modules and build artifacts. The `.geminiignore` file provides additional control for sensitive files that should be hidden from the AI but kept in version control.

## Key Takeaways

- File context injection via `@` syntax is the primary mechanism for providing specific context
- Exploration tools reduce manual navigation friction when navigating unknown codebases
- Safety-first editing prioritizes diff review before applying any changes
- `.geminiignore` provides selective AI visibility for sensitive files
- Testing verification completes the workflow loop
