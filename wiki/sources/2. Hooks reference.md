# Hooks Reference

This document provides the technical specification for Gemini CLI hooks, including JSON schemas and API details for extending CLI behavior through event-driven scripts.

## Overview

The hooks system enables developers to intercept and modify CLI operations at key lifecycle points. Hooks communicate with the CLI via stdin/stdout JSON protocol, with execution results communicated through exit codes.

## Communication Protocol

| Stream | Purpose |
|--------|---------|
| **stdin** | Receives JSON input (session_id, transcript_path, cwd, hook_event_name, timestamp) |
| **stdout** | Must output only valid JSON response |
| **stderr** | Logs and error messages |

## Exit Code Semantics

| Exit Code | Meaning | Behavior |
|-----------|---------|----------|
| `0` | Success | stdout parsed as JSON; preferred for all logic |
| `2` | System Block | Action blocked; stderr used as rejection reason |
| Other | Warning | Non-fatal failure; CLI continues with warning |

## Hook Categories

### Tool Hooks

- **BeforeTool**: Fires before tool invocation for validation, security checks, and parameter rewriting
- **AfterTool**: Fires after tool execution for auditing, context injection, and output hiding

### Agent Hooks

- **BeforeAgent**: Fires after user submits prompt but before agent planning; enables prompt validation
- **AfterAgent**: Fires after model generates response; enables validation and automatic retries

### Model Hooks

- **BeforeModel**: Fires before LLM request; can modify request or provide synthetic responses
- **BeforeToolSelection**: Fires before tool selection; enables tool filtering (limited output options)
- **AfterModel**: Fires per response chunk during streaming; enables real-time redaction

### Lifecycle Hooks

- **SessionStart**: Fires on startup, resume, or clear; advisory only
- **SessionEnd**: Fires on exit or session clear; best effort execution
- **Notification**: Fires on system alerts; observability only
- **PreCompress**: Fires before history compression; advisory only

## Configuration Schema

Hooks are defined in `settings.json` within the `hooks` object:

- **matcher**: Regex (for tools) or exact string (for lifecycle) to filter hook execution
- **sequential**: Whether hooks in a group run one-by-one or in parallel
- **hooks**: Array of hook configurations

Each hook configuration requires `type: "command"` and a shell `command`, with optional `name`, `timeout`, and `description`.

## Key Behavioral Notes

- **BeforeTool denial**: Turn continues; reason sent to agent as tool error
- **AfterTool denial**: Hides real output; reason replaces it
- **AfterAgent denial**: Forces retry with reason as new prompt
- **Exit Code 2 semantics vary**: Same exit code means different things depending on hook type
- **BeforeToolSelection limitations**: Cannot use decision/continue/systemMessage fields
- **Notification and PreCompress hooks**: Cannot block operations despite being "hooks"

## Stable Model API

The hooks system uses SDK-agnostic `LLMRequest` and `LLMResponse` structures to ensure compatibility across updates.

**LLMRequest**: Contains model, messages array (with role and content), config (temperature and generation params), and toolConfig.

**LLMResponse**: Contains candidates array (with content and finishReason) and usageMetadata.
