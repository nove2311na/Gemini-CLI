# Tool Hooks

Tool hooks intercept tool execution at two points: before invocation (for validation and modification) and after execution (for auditing and result manipulation).

## BeforeTool

Fires before a tool is invoked, enabling argument validation, security checks, and parameter rewriting.

### Trigger Configuration

Matchers support regex patterns for built-in tools (e.g., `read_.*` matches all file reading tools) and MCP tools following the `mcp_<server_name>_<tool_name>` naming convention.

### Available Input Fields

| Field | Type | Description |
|-------|------|-------------|
| `tool_name` | string | Name of the tool being called |
| `tool_input` | object | Raw arguments generated by the model |
| `mcp_context` | object | Optional metadata for MCP-based tools |
| `original_request_name` | string | Original tool name if this is a tail tool call |

### Response Options

- **decision: "deny"**: Prevents execution; `reason` sent to agent as tool error. Turn continues.
- **hookSpecificOutput.tool_input**: Object that merges with and overrides model's arguments
- **continue: false**: Kills entire agent loop immediately

### Exit Code 2 Behavior

Blocks tool execution. stderr is used as the rejection reason sent to the agent. The turn continues, allowing the agent to respond or retry.

### Example Use Cases

- **Argument Validation**: Verify file paths exist before read operations
- **Security Checks**: Block shell commands containing dangerous flags
- **Parameter Rewriting**: Normalize paths or inject credentials
- **Cost Control**: Track expensive tool usage

## AfterTool

Fires after a tool executes, enabling result auditing, context injection, and output manipulation.

### Available Input Fields

| Field | Type | Description |
|-------|------|-------------|
| `tool_name` | string | Name of the executed tool |
| `tool_input` | object | Original arguments passed to tool |
| `tool_response` | object | Result containing llmContent, returnDisplay, and optional error |
| `mcp_context` | object | Optional metadata for MCP-based tools |
| `original_request_name` | string | Original tool name if this is a tail tool call |

### Response Options

- **decision: "deny"**: Hides real output; `reason` replaces tool result
- **hookSpecificOutput.additionalContext**: Text appended to tool result for agent
- **hookSpecificOutput.tailToolCallRequest**: Execute another tool; result replaces original response
- **continue: false**: Kills entire agent loop immediately

### Exit Code 2 Behavior

Hides tool result from agent. stderr is used as replacement content. The turn continues.

### Example Use Cases

- **Auditing**: Log all tool executions with arguments and results
- **Context Injection**: Add derived information to tool results
- **Output Filtering**: Remove sensitive data before agent sees results
- **Programmatic Routing**: Chain tool calls where result of one becomes input of another

## Tail Tool Calls

The `tailToolCallRequest` feature enables programmatic tool routing. A hook can request execution of a follow-up tool whose result will replace the original tool's response.

```json
{
  "hookSpecificOutput": {
    "tailToolCallRequest": {
      "name": "format_file",
      "args": { "path": "${result.path}", "style": "prettier" }
    }
  }
}
```

This pattern is powerful for:
- Automatic formatting after code generation
- Validation after file modifications
- Post-processing tool outputs
- Chaining related operations

## Security Considerations

Tool hooks execute with the same permissions as the CLI itself. Consider:
- Timeout limits to prevent runaway scripts
- Matcher specificity to limit hook scope
- Exit code handling for partial failures
- Stderr logging for debugging without affecting behavior
