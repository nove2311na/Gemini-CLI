# Trusted Folders

The Trusted Folders feature is a security setting that gives you control over which projects can use the full capabilities of Gemini CLI. It prevents potentially malicious code from running by asking you to approve a folder before the CLI loads any project-specific configurations from it.

## Purpose

Trusted Folders protects your system by:

- Requiring explicit approval before loading project-specific configurations
- Preventing potentially malicious code from executing via custom configurations
- Giving you granular control over which workspaces can access full CLI capabilities

The feature scans folders during a **discovery phase** and displays:

| Category | Description |
|----------|-------------|
| **Commands** | Custom `.toml` command definitions that add new functionality |
| **MCP Servers** | Configured Model Context Protocol servers the CLI will connect to |
| **Hooks** | System or custom hooks that can intercept/modify CLI behavior |
| **Skills** | Local agent skills providing specialized capabilities |
| **Setting overrides** | Project-specific configurations overriding global user settings |

The dialog also highlights **Security Warnings** (e.g., auto-approved tools, disabled sandbox) and **Discovery Errors** (e.g., malformed `settings.json`).

## How to Enable

The Trusted Folders feature is **disabled by default**. To enable it:

1. Open your user `settings.json` file
2. Add the following configuration:

```json
{
  "security": {
    "folderTrust": {
      "enabled": true
    }
  }
}
```

Once enabled, the first time you run Gemini CLI from a folder, a trust dialog will automatically appear.

## Trust Levels

When the trust dialog appears, you have three options:

| Trust Level | Description | Use Case |
|-------------|-------------|----------|
| **Trust folder** | Grants full trust to the current folder (e.g., `my-project`) | Single isolated projects; granular control for sensitive directories |
| **Trust parent folder** | Grants trust to the parent directory (e.g., `safe-projects`), automatically trusting all subdirectories | When you keep all safe projects in one place |
| **Don't trust** | Marks the folder as untrusted; CLI operates in restricted "safe mode" | Untrusted or unknown projects |

Your choice is saved in `~/.gemini/trustedFolders.json`, so you're only asked once per folder.

### Impact of Untrusted Workspace

When a folder is **untrusted**, Gemini CLI runs in restricted mode with these features disabled:

1. **Workspace settings ignored**: `.gemini/settings.json` not loaded
2. **Environment variables ignored**: `.env` files not loaded
3. **Extension management restricted**: Cannot install/update/uninstall extensions
4. **Tool auto-acceptance disabled**: Always prompted before tool execution
5. **Automatic memory loading disabled**: No auto-loading files into context
6. **MCP servers do not connect**: No Model Context Protocol server connections
7. **Custom commands not loaded**: `.toml` command files ignored

Granting trust unlocks full CLI functionality for that workspace.

## Managing Your Trust Settings

### Change Current Folder's Trust
Run the `/permissions` command within the CLI to reopen the interactive dialog and change the trust level for the current folder.

### View All Trust Rules
Inspect the contents of `~/.gemini/trustedFolders.json` in your home directory to see all trusted and untrusted folder rules.

### Override Trust File Location
By default, trust settings are saved to `~/.gemini/trustedFolders.json`. To use a different location, set the environment variable:

```bash
export GEMINI_CLI_TRUSTED_FOLDERS_PATH=/path/to/custom/trustedFolders.json
```

### Headless/Automated Environments

In CI/CD pipelines where interactive prompts aren't possible, the CLI throws `FatalUntrustedWorkspaceError` if the folder is untrusted. Bypass options:

- **Command-line flag**: `--skip-trust`
- **Environment variable**: `GEMINI_CLI_TRUST_WORKSPACE=true`

These trust the current workspace for the session without prompting.

## The Trust Check Process (Advanced)

Trust is determined in this order:

1. **IDE trust signal**: If using IDE Integration, the CLI first asks the IDE if the workspace is trusted (highest priority)
2. **Local trust file**: If IDE is not connected, the CLI checks `~/.gemini/trustedFolders.json`
