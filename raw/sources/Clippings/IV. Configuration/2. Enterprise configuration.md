# Gemini CLI for the enterprise

This document outlines configuration patterns and best practices for deploying and managing Gemini CLI in an enterprise environment. By leveraging system-level settings, administrators can enforce security policies, manage tool access, and ensure a consistent experience for all users.

> **Caution:** These measures are designed to prevent accidental misuse and enforce corporate policy in a managed environment, not to defend against a malicious actor with local administrative rights.

## Centralized configuration: The system settings file

The most powerful tools for enterprise administration are the system-wide settings files. Settings are merged from four files in this precedence order:

1.  **System Defaults** (`system-defaults.json`)
2.  **User Settings** (`~/.gemini/settings.json`)
3.  **Workspace Settings** (`<project>/.gemini/settings.json`)
4.  **System Overrides** (`settings.json`)

The **System Overrides** file has the final say for single-value settings. Arrays and objects are merged.

### Location:
*   **Linux:** `/etc/gemini-cli/settings.json`
*   **Windows:** `C:\ProgramData\gemini-cli\settings.json`
*   **macOS:** `/Library/Application Support/GeminiCli/settings.json`

The path can be overridden using the `GEMINI_CLI_SYSTEM_SETTINGS_PATH` environment variable.

## Enforcing system settings with a wrapper script

To prevent users from bypassing the centrally managed configuration, enterprises can deploy a wrapper script that ensures the environment variable is always set.

**Example wrapper script (`/usr/local/bin/gemini`):**
```bash
#!/bin/bash
export GEMINI_CLI_SYSTEM_SETTINGS_PATH="/etc/gemini-cli/settings.json"
REAL_GEMINI_PATH=$(type -aP gemini | grep -v "^$(type -P gemini)$" | head -n 1)
exec "$REAL_GEMINI_PATH" "$@"
```

## User isolation in shared environments

In shared compute environments, you can isolate Gemini CLI state by overriding the user's home directory using the `GEMINI_CLI_HOME` environment variable.

```bash
# Isolate state for a specific job
export GEMINI_CLI_HOME="/tmp/gemini-job-123"
gemini
```

## Restricting tool access

### Allowlisting with coreTools
The most secure approach is to explicitly add approved tools to an allowlist.
```json
{
  "tools": {
    "core": ["ReadFileTool", "GlobTool", "ShellTool(ls)"]
  }
}
```

### Disabling YOLO mode
Force all tool executions to require user confirmation.
```json
{
  "security": {
    "disableYoloMode": true
  }
}
```

## Managing custom tools (MCP servers)

### Enforcing a catalog of tools
To create a secure, centrally-managed catalog of tools, the system administrator should:
1.  Define the full configuration for every approved server in the `mcpServers` object.
2.  Add the names of those servers to an allowlist using the `mcp.allowed` setting.

**Example system `settings.json`:**
```json
{
  "mcp": {
    "allowed": ["corp-data-api", "source-code-analyzer"]
  },
  "mcpServers": {
    "corp-data-api": {
      "command": "/usr/local/bin/start-corp-api.sh",
      "timeout": 5000
    },
    "source-code-analyzer": {
      "command": "/usr/local/bin/start-analyzer.sh"
    }
  }
}
```

## Enforcing sandboxing for security

Force all tool execution to happen within a containerized environment.
```json
{
  "tools": {
    "sandbox": "docker"
  }
}
```

## Telemetry and auditing

Configure Gemini CLI to send telemetry data to a central location for auditing.
```json
{
  "telemetry": {
    "enabled": true,
    "target": "gcp",
    "logPrompts": false
  }
}
```
*Note: Set `logPrompts` to `false` to avoid collecting sensitive information.*

## Authentication

Enforce a specific authentication method (e.g., Google OAuth) for all users.
```json
{
  "security": {
    "auth": {
      "enforcedType": "oauth-personal"
    }
  }
}
```

### Restricting logins to corporate domains
For Google Workspace, configure your web proxy to add the `X-GoogApps-Allowed-Domains` HTTP header to requests to `google.com`.
