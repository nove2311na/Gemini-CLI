# Gemini CLI Project Setup & Best Practices Research

This document summarizes effective file and folder configurations for working with Gemini CLI, based on research conducted on May 5, 2026.

## 1. Context Hierarchy (GEMINI.md)
Gemini CLI uses a hierarchical system to source context. It loads context files from multiple levels:
- **Global Context:** `~/.gemini/GEMINI.md` (for user-wide preferences).
- **Project Root Context:** `./GEMINI.md` (for project-wide architecture, conventions, and workflows).
- **Subdirectory Context:** `./src/GEMINI.md` (for scoped instructions specific to a module or directory).

**Tip:** Use these files to define "Rules of Engagement," architectural standards, and team-shared workflows.

## 2. Recommended Directory Structure
A typical "Gemini CLI–friendly" project might look like this:
```text
project-root/
├── GEMINI.md (Main project context)
├── .gemini/
│   ├── settings.json (Project-specific CLI settings)
│   ├── subagents/ (Custom specialized agents)
│   └── hooks/ (Workflow automation scripts)
├── .geminiignore (Exclude files from context)
├── docs/ or Clippings/ (Project documentation)
├── src/ (Source code)
│   └── GEMINI.md (Source-specific context)
└── tests/ (Test files)
```

## 3. Key Best Practices
- **Documentation as Context:** Store documentation in a dedicated folder (like `Clippings/` or `docs/`). This allows agents to quickly reference the "Source of Truth" without scanning unrelated code.
- **Custom Commands & Hooks:** Use the `hooks` feature to automate tasks, inject dynamic context (like security scans), or enforce policies.
- **Memory Management:** Use the `MEMORY.md` file (stored privately) for personal project-specific notes that shouldn't be committed to the repository.
- **Agent Awareness:** Create specialized subagents in `.gemini/subagents/` to handle specific tasks like "Test Runner," "Documenter," or "Architect."
